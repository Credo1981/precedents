Формируем портфолио на GitHub

Как правильно оформить Open Source проект на GitHub
  • В каждом репозитории должен быть .gitignore. Общее правило: код — который генерируется в процессе разработки, тестирования, сборки/компиляции, рантайма — должен находиться в gitignore.
  • Github, при создании проекта, позволяет выбрать подходящую лицензию. Файл должен называться LICENSE, без расширения. Очень подробная статья на тему выбора свободной лицензии
  • README.md.
    1. Назначение библиотеки (проекта, инструмента, фреймворка)
    2. Системные требования (версия языка, требования к ресурсам, системные зависимости, нужные расширения)
    3. Шаги по установке, сборке, запуску
    4. Примеры использования или ссылки на документацию
  • Версионирование (SemVer) и журнал изменений (CHANGELOG.md)
    • http://semver.org/
    • http://keepachangelog.com
  • Тесты — единственный формальный способ доказать корректность и работоспособность кода
  • Travis-ci подключается к github в пару кликов. Наличие .travis.yml в репозитории — обязательно,
  • Пакетный менеджер
    • Если для вашей платформы есть пакетный менеджер, потрудитесь чтобы ваш код мог быть получен через него. Composer, npm, maven, etc
    • Также важно чтобы актуальная версия (в соответствии с SemVer, ага) автоматически, после прохождения CI (и тестирования, ага), попадала в реестр.
    • К счастью, для большинства мейнстримовых языков такие менеджеры есть и интегрируются с github / travis в пару кликов / строчек конфига.
    • Зависимости вашего кода, само собой должны быть надлежащим образом оформлены при помощи файла конфигурации того же менеджера.
  • Внедрение зависимостей, SOLID, etc
  • Используйте стандарты языка и его сообщества: JSR, PSR, общепринятые code styles.
  • Подробная документация, примеры, FAQ
    • Wiki-документация. Может быть размещена на том же github, либо в виде набора md файлов в самом репозитории или на github pages, или вообще на отдельном сайте.
    • Небольшие проекты-репозитории с типичными примерами использования. Бывает очень полезно для проектов, представляющих собой что-то среднее между фреймворком и библиотекой.
    • FAQ — пополняемый из собственного и пользовательского опыта
    • Для реактивной поддержки github имеет issues
  • Файлы CONTRIBUTING.md, CONTRIBUTORS.md, ISSUE_TEMPLATE.md, PULL_REQUEST_TEMPLATE.md
  • Пример четлиста: https://github.com/github/gitignore/community
  • Бейджи
    • процент покрытия
    • актуальная стабильная версия
    • статистика скачиваний в реестре / звезд, или другая фаллометрия свидетельствующая о популярности или востребованности
  • Интеграция с сервисами
    • https://codecov.io
    • https://codeclimate.com/
    • https://www.bithound.io
    • https://scrutinizer-ci.com/
    • https://codeship.com/
  • Docker

Перечень законов логики

Закон двойного отрицания
!!A = A
Закон Де Моргана

для логического сложения

!(A || B) = !A & !B

для логического умножения

!(A & B) = !A || !B
Закон исключения констант

для логического сложения

A || 1 = 1; A || 0 = A

для логического умножения

A & 1 = A; A & 0 = 0
Закон исключённого третьего
A || !A = 1
Закон противоречия
A & !A = 0
Закон сочетательный (ассоциативности)

для логического сложения

(A || B) || C = A || (B || C)

для логического умножения

(A & B) & C = A & (B & C)
Закон распределительный (дистрибутивный)

для логического сложения

(A || B) & C = (A & C) || (B & C)

для логического умножения

(A & B) || C = (A || C) & (B || C)
Закон переместительный (комутативный)

для логического сложения

A || B = B || A

для логического умножения

A & B = B & A
Закон поглащения

для логического сложения

A || (A & B) = A

для логического умножения

A & (A || B) = A
Закон имподентности (равносильности)

для логического сложения

A || A = A

для логического умножения

A & A = A
